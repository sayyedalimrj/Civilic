# Civilic — جهت محصول (Product Steering)

Civilic یک **سامانه‌ی مدیریت پروژه و گردش‌کار صورت‌وضعیت** برای پروژه‌های عمرانی/پیمانکاری ایران است. این یک «نمایش‌گر جدول تکسا» نیست؛ تکسا فقط **لایه‌ی سازگاری import/export** است.

## اصول بنیادین
- Civilic **فارسی‌محور و RTL‌محور** است. تمام UI فارسی و راست‌به‌چپ.
- Civilic یک **محصول گردش‌کار** است، نه داشبورد BI. هیچ نمودار mock، عدد تصادفی یا KPI تزئینی.
- **داده‌ی خام تکسا از کاربر عادی پنهان است.** نام جدول‌های `brv_*`/`base_*` هرگز در UI اصلی دیده نمی‌شود؛ فقط در «تنظیمات → پیشرفته → داده خام تکسا» با مجوز `texsa.raw.view`.
- سه طرف اصلی: **کارفرما (EMPLOYER)**، **مشاور/دستگاه نظارت (CONSULTANT)**، **پیمانکار (CONTRACTOR)**. هر طرف کاربران، نقش‌ها و کانال داخلی خود را دارد.
- **هر اقدام مهم باید سمت سرور permission-check شود.** مخفی‌کردن دکمه کافی نیست.

## دو لایه‌ی محصول
1. **لایه‌ی سازگاری تکسا:** حفظ بدون‌تلفات داده‌ی خام، schema، import، normalize، export، گزارش round-trip.
2. **لایه‌ی گردش‌کار Civilic:** UI تمیز فارسی، کاربران/نقش‌ها/سازمان‌ها، گردش‌کار پروژه، صورت‌وضعیت، اسناد، چت، مکاتبات، گزارش.

## چرخه‌ی کامل تکسا
```
Important project.svzt → parse → preserve raw → normalize → نمایش در UI تمیز
→ ادامه‌ی گردش‌کار → export XML سازگار تکسا → گزارش round-trip
```

## قاعده‌ی طلایی داده
- داده‌ی خام تکسا **هرگز** حذف نمی‌شود (mirror + rawJson + raw row).
- هر موجودیت نرمال‌شده اشاره‌گری به import مبدأ و raw row خود نگه می‌دارد.
- رکوردهای ساخته‌شده در Civilic پس از import با `source = CIVILIC` علامت می‌خورند.

## معیار پذیرش محصول (خلاصه)
کاربر سه نقش (کارفرما/مشاور/پیمانکار) باید بتواند وارد شود، پروژه‌ی واقعی خاتم را ببیند، صورت‌وضعیت بسازد/ارسال/بررسی/برگشت/تأیید/قفل کند، چت گروهی و کانال داخلی داشته باشد، مکاتبه و سند ایجاد کند، تعدیل را پردازش کند، و خروجی سازگار تکسا + گزارش round-trip بگیرد. build/typecheck/lint سبز و آماده‌ی Vercel با PostgreSQL.
